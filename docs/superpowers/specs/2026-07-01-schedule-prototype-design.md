# 일정 조율 프로토타입 설계

## 목표
토스 Product Designer Challenge 2026 과제에서 정의한 문제를 작동하는 웹 프로토타입으로 증명한다.

프로토타입은 예쁜 캘린더 앱이 아니라, 주최자가 충돌하는 조건 속에서 모두에게 설명 가능한 1시간을 확정할 수 있는지를 보여준다.

## 문제 정의
6명 회의 일정 조율의 진짜 문제는 빈 시간을 찾는 것이 아니다.

주최자가 필수 조건과 개인별 양보를 한눈에 이해해, 모두에게 설명 가능한 1시간을 자신 있게 확정하지 못하는 것이다.

## 프로토타입이 증명할 것
- 사용자가 조건을 쉽게 입력할 수 있다.
- 불가능한 시간과 피하고 싶은 시간이 구분된다.
- 필수 참석자와 선택 참석자가 다르게 반영된다.
- 시스템이 추천 1시간을 제시한다.
- 추천 이유가 짧고 납득 가능하다.
- 주최자가 확정하고 공유할 수 있다.

## 핵심 흐름
1. 회의 만들기
2. 6명 조건 입력
3. 추천 결과
4. 확정/공유

## 화면 설계

### 1. 회의 만들기
목적은 사용자가 문제 상황을 바로 이해하고 회의 조건을 만드는 것이다.

화면에 보여줄 정보:
- 회의명: 다음 주 제품 리뷰
- 시간: 1시간
- 범위: 다음 주
- 참석자 6명
- 필수 참석자 4명, 선택 참석자 2명

주요 행동:
- `조건 입력하기`

### 2. 6명 조건 입력
목적은 각 사람의 조건 차이를 단순하게 보여주는 것이다.

각 참석자 카드에는 다음 정보만 둔다.
- 이름
- 참석 중요도: 필수 또는 선택
- 불가능한 시간
- 피하고 싶은 시간

입력은 실제 편집 가능한 복잡한 캘린더가 아니라, 샘플 조건을 선택된 상태로 보여주는 방식으로 구현한다. 이 과제의 핵심은 입력 도구가 아니라 조건이 추천 결과에 반영되는 구조를 증명하는 것이다.

주요 행동:
- `추천 시간 보기`

### 3. 추천 결과
가장 중요한 화면이다.

추천 1시간을 가장 크게 보여준다. 추천 아래에는 확정 근거를 짧게 보여준다.

추천 예시:
- 화요일 오후 3:00

근거 예시:
- 필수 참석자 전원 가능
- 선택 참석자 1명 어려움
- 선호 충돌 최소

보조 정보:
- 왜 다른 시간보다 나은지 한 줄로 설명한다.
- 필요한 경우에만 2순위 후보를 펼쳐볼 수 있다.

주요 행동:
- `이 시간으로 확정`

### 4. 확정/공유
목적은 주최자가 확정 결과와 공유 문구를 바로 확인하는 것이다.

화면에 보여줄 정보:
- 확정된 시간
- 참석 가능 요약
- 공유 문구

주요 행동:
- `공유 문구 복사`
- `다시 보기`

## 추천 기준
추천 로직은 점수를 과시하지 않는다. 내부 계산은 단순하게 둔다.

우선순위:
1. 필수 참석자의 불가능한 시간이 없는 후보를 우선한다.
2. 선택 참석자의 불가능한 시간이 적은 후보를 우선한다.
3. 피하고 싶은 시간이 적은 후보를 우선한다.
4. 동률이면 더 이른 후보를 선택한다.

사용자에게는 점수 대신 결정 근거로 보여준다.

## 시각 방향
- 밝은 배경과 높은 대비를 사용한다.
- 추천 결과 화면의 정보 위계를 가장 강하게 만든다.
- 장식보다 버튼, 상태 배지, 요약 문장의 정렬을 다듬는다.
- 색은 의미 전달에만 쓴다.
- 화면은 모바일 폭에서도 먼저 읽히게 만든다.

## 구현 범위
- Vite + React + TypeScript
- 백엔드 없음
- 샘플 데이터 기반
- 로컬 상태만 사용
- 테스트는 추천 로직에 집중

## 하지 않을 것
- 전체 일정 관리 앱으로 확장하지 않는다.
- 복잡한 캘린더 매트릭스를 만들지 않는다.
- 후보 시간을 많이 나열하지 않는다.
- AI나 Council을 제품 컨셉처럼 드러내지 않는다.
- 점수표로 사용자를 설득하지 않는다.
