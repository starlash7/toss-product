# 문제 정의 검증

## 목적
토스 Product Designer Challenge 2026의 일정 조율 과제에서 문제 정의와 솔루션 방향이 평가 기준에 맞는지 검증했다.

Council은 제품 화면에 드러내지 않고, 문제 정의를 여러 관점에서 검증하는 과정으로만 사용한다.

## 검증한 문제 정의

> 6명 일정 조율의 본질은 모두가 완벽히 원하는 시간을 찾는 것이 아니라, 필수 제약을 지키면서 양보 비용이 가장 작은 1시간을 납득 가능하게 선택하는 것이다.

## Council 핵심 결론

현재 방향은 본질에 가깝다. 다만 `양보 비용`이라는 표현은 설계자 언어처럼 보일 수 있다.

수상권으로 가려면 문제를 "최적 시간 계산"이 아니라 "주최자가 불완전한 결정을 설명 가능하게 확정하지 못하는 문제"로 잡아야 한다.

## 최종 문제 정의

> 6명 회의 일정 조율의 진짜 문제는 빈 시간을 찾는 것이 아니다.  
> 주최자가 필수 조건과 개인별 양보를 한눈에 이해해, 모두에게 설명 가능한 1시간을 자신 있게 확정하지 못하는 것이다.

화면과 제출 답변에서는 더 부드럽게 표현한다.

> 모두가 완벽히 원하는 시간은 없어도, 모두가 받아들일 수 있는 가장 덜 무리한 1시간은 찾을 수 있어야 한다.

## 반복해서 나온 약점
- `양보 비용`이 화면에서 보이지 않으면 말만 그럴듯해 보인다.
- `납득 가능`이 추상적이면 설득력이 약하다.
- 6명이라는 조건에서 생기는 판단 부담이 더 드러나야 한다.
- 주최자의 판단과 설명 부담이 문제 정의의 중심에 있어야 한다.
- AI 추천이나 점수 계산처럼 보이면 토스식 단순함에서 멀어진다.

## UX 원칙
- 불가능한 시간과 피하고 싶은 시간을 분리한다.
- 필수 참석자와 선택 참석자의 조건을 다르게 반영한다.
- 후보를 많이 보여주지 않고 추천 1개를 먼저 보여준다.
- 계산식이 아니라 결정에 필요한 근거만 보여준다.
- 화면 흐름은 `조건 입력 → 추천 1시간 → 이유 → 확정`으로 간다.

## 프로토타입 필수 화면
- 6명의 조건 입력
- 필수 참석자와 선택 참석자 구분
- 불가능한 시간과 피하고 싶은 시간 구분
- 추천된 1시간
- 추천 이유: `필수 참석자 전원 가능`, `선택 참석자 1명 어려움`, `선호 충돌 최소`
- 확정과 공유 화면

## 하지 않을 것
- Council 느낌을 제품 화면에 드러내지 않는다.
- 복잡한 캘린더 매트릭스를 만들지 않는다.
- AI나 최적화 알고리즘을 과시하지 않는다.
- 후보 시간을 많이 나열해 사용자가 다시 비교하게 만들지 않는다.
- 점수표로 사용자를 설득하려 하지 않는다.
