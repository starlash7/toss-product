# 지원서 답변 초안

## 작성 기준
- 지원서 문항은 한국어로 작성한다.
- 각 문항은 공백 포함 700자 이하로 유지한다.
- 문제 정의는 "후보 부족"이 아니라 "확정 근거 부족"을 중심에 둔다.
- 솔루션은 추천 로직보다 사용자가 보는 화면 정보 위계를 먼저 보여준다.
- Council은 문제 검증 과정으로만 활용하고, 지원서 본문에는 드러내지 않는다.

## 1. 사용자의 어떤 문제를 발견했나요? 그것이 왜 문제라고 생각했나요?

회의 일정 조율의 본질은 빈 시간 탐색이 아니라, 충돌하는 조건 속에서 확정 기준을 만드는 일이라고 봤습니다. 6명 회의를 잡는 주최자는 필수 참석자, 선택 참석자, 개인 선호를 한꺼번에 비교해야 합니다. 누군가는 꼭 참석해야 하고, 누군가는 점심 직후를 피하고 싶고, 누군가는 특정 요일에 외근이 많습니다. 투표형 도구나 캘린더 매트릭스는 가능한 시간을 보여주지만, 어떤 조건을 우선해야 하는지까지 결정해주지는 않습니다. 그래서 주최자는 가능한 시간을 고르는 사람이 아니라, 조건이 충돌할 때 어떤 기준으로 확정했는지 설명해야 하는 사람이 됩니다. 제가 정의한 문제는 시간 부족이 아니라, 주최자가 선택의 책임을 혼자 해석하고 설명해야 하는 구조입니다.

## 2. 그 문제를 어떻게 해결했나요?

입력은 참석자가 쉽게 답할 수 있게 세 가지로 줄였습니다. 절대 불가능한 시간, 가능하지만 피하고 싶은 시간, 그리고 필수 참석자인지 선택 참석자인지입니다. 결과 화면은 주최자가 바로 확정할 수 있도록 “추천 시간 → 확정 가능 여부 → 근거 요약 → 대안 보기” 순서로 구성했습니다. 추천 카드에는 가장 괜찮은 1시간을 크게 보여주고, 바로 아래에 “필수 참석자 전원 가능”, “선택 참석자 1명 어려움”, “선호 충돌 최소” 같은 근거 배지를 배치합니다. 기본값은 추천 1개지만, 납득이 어려울 때만 2순위 후보와 충돌 상세를 펼쳐볼 수 있게 했습니다. 이 UX의 목적은 더 많은 후보가 아니라, 한 번에 확정해도 되는 이유를 먼저 이해하게 만드는 것입니다.

## 3. 왜 이 방식으로 설계했나요?

이 방식은 일정 조율의 핵심을 탐색이 아니라 확정으로 봤기 때문에 선택했습니다. 투표형 방식은 많은 사람이 고른 시간을 찾기 쉽지만, 꼭 참석해야 하는 사람과 선택 참석자의 차이를 놓치기 쉽습니다. 전체 캘린더 매트릭스는 익숙하지만, 주최자가 다시 겹치는 시간을 해석해야 합니다. 그래서 불가능한 조건과 피하고 싶은 조건을 분리하고, 필수 참석 여부를 우선순위로 반영했습니다. 화면도 탐색용 캘린더가 아니라 “이 시간으로 정해도 되는가”를 판단하는 의사결정 화면으로 설계했습니다. 범위는 일정 관리 전체가 아니라 확정 순간의 판단 비용을 줄이는 핵심 흐름에 집중했습니다.
